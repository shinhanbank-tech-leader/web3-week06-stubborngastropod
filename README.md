## 레포트 내용

- 스마트컨트랙트

스마트컨트랙트는 이미 설정된 프로그램에 따라 특정 조건이 충족되면 그에 따른 동작이 수행되는 프로토콜입니다. 트랜잭션과 인풋에 따라 결과값이 좌우된다는 점에서 하나의 복잡한 함수, 즉 우리가 일상에서 사용하는 애플리케이션과도 유사하다고 볼 수 있습니다.

스마트컨트랙트가 기존의 프로그램과 다른 점은 특정 기관이나 개인이 아닌 프로그램 자체에 대한 신뢰를 기반으로 거래가 일어난다는 점입니다. DB나 소스코드에 대한 접근이 철저히 제한되어있기 때문에 조작이나 부정행위가 일어나지 않는다는 보장을 받을 수 있습니다.

스마트컨트랙트에 대한 신뢰는 어떻게 형성되는지에 대한 과제가 남는데, 이는 Etherscan 등의 사이트에 개발자가 본인이 deploy한 스마트컨트랙트의 소스코드를 공개하는 식으로 이루어집니다. 소스코드의 무결성을 보장받으면서도, 거래의 정합성은 사용자가 직접 확인하고 거래가 일어날 수 있도록 합니다.

스마트컨트랙트는 우리가 일반적으로 사용하는 어플리케이션과 같이 코딩 언어로 구성된 프로그램이면서도, 그 자체가 만천하에 내용이 공개된 강력한 계약서라고 볼 수 있습니다.


- 은행 시스템과의 차이

기존 은행 시스템은 서버 상에서 데이터 저장과 연산 처리가 이루어지기 때문에 관련된 제반 시설과 제도 등의 기반이 필요합니다. 데이터센터 및 서버 구축과 더불어 이를 유지보수하고 관리하기 위해 서버 비용이 수반됩니다.

블록체인 시스템은 별도의 서버가 없지만 검증과 데이터 저장 역할을 블록체인 노드로 참여하는 PC가 담당합니다. 블록체인에 트랜잭션을 올리기 위해서 해당 컴퓨팅 자원을 사용하기 때문에 가스비가 소모됩니다.

가스비는 거래의 특성에 따라 천차만별로 달라질 수 있는데, 그렇기 때문에 스마트컨트랙트 코드를 작성할 때에는 가스비를 고려하여 최대한 효율적으로 코딩하는 것이 중요해집니다. 메모리, 상태, 접근 제어 등의 코딩 규칙이 기존 시스템에서는 비즈니스 로직 측면에서의 완성도를 위해 사용되었다면 스마트컨트랙트에서는 비교적 함수의 비용적 특성에 중점을 두어 선언 시에 설정하는 경향이 있어 보입니다.


- 스마트컨트랙트의 기존 시스템 대체 가능성

스마트컨트랙트는 명시적인 조건이 곧 실행 요건이 되기 때문에 정량적으로 측정될 수 없거나 모호한 요소가 포함된 거래에는 사용될 수 없습니다. 대신 에스크로, 담보대출, 정산, 기관통신 등의 정량적인 거래는 이론상 스마트컨트랙트로 모두 대체될 수 있을 것입니다.

통상적인 은행 거래도 대부분이 정량화되어있는 기준에 의거하기 때문에 문제가 없을 것으로 보입니다. 다만 신뢰성을 위해 소스코드를 모두 공개해야하는 스마트컨트랙트의 특성상 일부 제한이 있을 수 있습니다. 예시로 대출 상품의 경우 당행에서 고객 유치와 수익성 향상에 크게 연관된 '전략'이라는 요소가 포함되어 있기 때문에 금리 산출 등의 전략과 관련해서는 일부 블랙박스 내에서 처리하는 등의 조치를 취할 수 있습니다.
